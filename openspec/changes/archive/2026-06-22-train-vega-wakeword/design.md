## Context

The Mac Ear (`apps/mac-ear/`) runs a continuous wake-word detector based on the openWakeWord ONNX stack: a shared `melspectrogram.onnx` + `embedding_model.onnx` pair (Google speech embedding) followed by one tiny per-keyword classifier head. Today, two community-trained heads (`Janet.onnx`, `edna.onnx`) ship in `Sources/VegaEar/Resources/` purely as development placeholders. The product wake word is "Вега" (Russian, /ˈvʲeɡə/), for which no head exists.

openWakeWord's training recipe trains only the per-keyword head (a small MLP / GRU over a window of 96-dim embeddings). The shared mel + embedding front-end is reused unchanged. Heads are typically a few hundred KB and train in minutes once embeddings are cached.

Hardware constraint: training will happen on the user's Apple Silicon mac. There is no CUDA available, but the head is small enough that CPU is enough; PyTorch's MPS backend is available as an option but not required. The widespread belief that "Apple does not let you train on a laptop" stems from the absence of NVIDIA GPUs, not any platform restriction.

Data the project needs to produce:
- Synthetic positives generated by Piper TTS using Russian voices (`ru_RU-*`).
- Real positives recorded by the primary user across distances, volumes, rooms, times of day.
- Negatives sourced from Common Voice Russian + ambient room recordings; openWakeWord's published "background audio" set is also reusable.

The Ear's runtime contract (protocol with Core, audio capture path, threshold UI) does not change — only the candidate model file and the default candidate name change.

## Goals / Non-Goals

**Goals:**

- A drop-in `Vega.onnx` classifier head that the existing `OpenWakeWordDetector` loads with zero implementation changes.
- A reproducible training pipeline under `tools/wake-training/` that any contributor can re-run on a clean macOS checkout in under ~2 hours (excluding dataset download time).
- Calibration data (precision/recall curve at varying thresholds) committed alongside the model so the default `Preferences.wakeThreshold` can be chosen with evidence, not a guess.
- "Вега" detection that is responsive (≤400 ms detection latency, matching the existing mac-ear spec) on the primary user's voice in normal home conditions, with false-positive rate well under 1 per hour of typical office/home audio.
- Test heads (`Janet.onnx`, `edna.onnx`) removed from the bundle.

**Non-Goals:**

- Multi-language support: only Russian "Вега" is in scope. An English "Vega" head can be a follow-up.
- Multi-user voice adaptation: the head is trained primarily for one user; generalization to other speakers is a bonus, not a requirement.
- Online / on-device fine-tuning: training is offline and developer-driven.
- Replacing the openWakeWord stack with Porcupine, Vosk, Apple SFSpeechRecognizer, or anything else.
- Automating recording capture inside the Mac Ear app. Recordings are done with QuickTime / `sox` / similar and dropped into the dataset directory by hand.
- Hosting datasets or trained models in this repo. Datasets are gitignored; the small `Vega.onnx` head is checked in as a binary asset (same as the existing `Janet.onnx`).

## Decisions

### Decision: Keep openWakeWord stack; train only the classifier head

**Choice:** Train a new head against the unchanged `embedding_model.onnx` already shipped.

**Why:** The shared front-end is what makes openWakeWord cheap. Replacing it would mean recomputing every embedding, re-bundling a ~1 MB model, and reverifying the Swift pipeline. The head trains in minutes; the front-end is already proven in production.

**Alternatives considered:**
- *Porcupine (Picovoice).* Has a custom-keyword console + supports Russian. Rejected: per-user license fee for commercial use, swap of the whole detector implementation, vendor lock-in. The project already absorbed the openWakeWord integration cost.
- *Vosk KWS.* Free, supports Russian. Rejected: model size and latency are worse, and it's a fundamentally different runtime (full STT engine repurposed for keyword spotting).
- *Apple SFSpeechRecognizer offline keyword spotting.* Rejected: not designed for always-on background detection, requires `requestAuthorization`, and the API is more opaque than ONNX inference.

### Decision: Single head, Russian only, candidate name "Vega"

**Choice:** Ship one `Vega.onnx` head trained on Russian "Вега" pronunciations; default `candidateNames` becomes `["Vega"]`.

**Why:** Mixing English and Russian utterances in one head's training set hurts the head's specificity (the same head must fire for both `/ˈveɪɡə/` and `/ˈvʲeɡə/`). The runtime already supports multiple heads, so an English head can be added later as a parallel candidate without retraining the Russian one. For MVP we want the head that the primary user actually says.

The file is named `Vega.onnx` (ASCII) so it matches the existing English-letter naming convention in `Resources/` and avoids any Cyrillic-in-filename surprises in build tooling. The `candidateNames` entry is `"Vega"` — also ASCII — and is what shows up in detection logs.

**Alternatives considered:**
- *Two heads, ["Vega", "Вега"], both shipped from day one.* Rejected for MVP: doubles training work, doubles latency per frame, and the English head has no real consumer yet.
- *Single multilingual head.* Rejected: in practice this raises FPR on similar-sounding Russian words ("Бега", "Мега") without buying much.

### Decision: Synthetic + real positives, in roughly 4:1 ratio

**Choice:** Training set per round: ~2000 Piper-generated utterances spanning all available `ru_RU-*` voices, mixed with ~50–150 real user recordings, then augmented (additive noise, reverb, gain, time-shift) into ~10k positive examples.

**Why:** Pure synthetic positives generalize broadly but miss the user's mic and acoustic signature. Pure real recordings overfit fast (50 examples is too small for a head to generalize well). The 4:1 split is the openWakeWord author's published recommendation and matches the project's reality where the user can record minutes, not hours.

**Alternatives considered:**
- *Synthetic-only.* Cheaper, but expected to undershoot recall on the user's voice.
- *Real-only.* Would require hours of recording; impractical.

### Decision: Negatives from Common Voice ru + openWakeWord background audio

**Choice:** Negatives = a fixed sample of Common Voice ru clips (a few hours total) + openWakeWord's published "background audio" dataset (rooms, TV, music) + a small set of "near-miss" words (e.g. "Бега", "Мега", "Вена") to harden the head.

**Why:** Negatives are what kill false-positive rate. The published background set is what other openWakeWord heads have been trained against — using it gives us comparable FPR characteristics. Adding near-miss negatives explicitly trains the head's decision boundary along the directions it is most likely to fail.

**Alternatives considered:**
- *Random YouTube audio.* Rejected: licensing and reproducibility.
- *Generated noise only.* Rejected: real speech in the background is what triggers false positives in practice.

### Decision: Embedding cache as on-disk parquet

**Choice:** Run the full positive + negative dataset through `embedding_model.onnx` once and persist 96-dim vectors + labels to a single parquet file per split (train/val/test). Training reads only this file; raw audio is not opened during training loops.

**Why:** Embedding inference is the slow step (~10× the head's training cost). Caching makes retraining a single change (e.g. different head architecture, different threshold sweep) take seconds. Parquet is fast to read in pandas/pytorch DataLoaders and is friendly for ad-hoc analysis.

**Alternatives considered:**
- *npy / pickle dumps.* Rejected: less ergonomic for label joins and threshold-sweep analysis.
- *Recompute embeddings each run.* Rejected: makes iteration painful.

### Decision: Train with PyTorch on CPU; MPS optional

**Choice:** Training script uses PyTorch. Default device is CPU; pass `--device mps` to use Apple GPU.

**Why:** The head is small (a few thousand parameters). CPU is fine and avoids the occasional MPS op-not-implemented errors that still surface on macOS. MPS as an opt-in keeps the door open if larger heads are experimented with.

**Alternatives considered:**
- *TensorFlow.* The original openWakeWord training scripts use TF; we deliberately step off that path because TF on Apple Silicon is rougher to set up than PyTorch.
- *Keras / scikit-learn.* Too small a library surface to justify; PyTorch is already the team's standard for Python-side ML work.

### Decision: Head architecture matches openWakeWord defaults

**Choice:** Same head topology as Janet/edna: a small MLP/GRU over 16 consecutive 96-dim embeddings, sigmoid output. Concretely, mirror the published openWakeWord head graph so `OpenWakeWordDetector.detectClassifierWindow` continues to work.

**Why:** Detector code in `OpenWakeWordDetector.swift` assumes `windowFrames` defaults to 16 and the output is a single scalar in [0, 1]. Diverging from this means Swift-side changes and re-spec. Staying on the published topology gives "drop-in compatibility" for free.

### Decision: Calibration as a script that emits a report

**Choice:** `tools/wake-training/eval_threshold.py` runs the trained head against a held-out test set and a "long background" recording (≥1 hour of typical environment audio), then emits a markdown report with precision/recall at thresholds {0.3, 0.4, 0.5, 0.6, 0.7, 0.8, 0.9} and false-positives-per-hour. Report is committed to `tools/wake-training/reports/<timestamp>.md`.

**Why:** Picking a default threshold without numbers is guesswork. A committed report makes the chosen default auditable and gives future-us a baseline if we retrain.

### Decision: Retain runtime contract

**Choice:** No changes to `OpenWakeWordDetector` API, no changes to `Preferences`, no changes to protocol or socket events. Only the resource file set and the default `candidateNames` value change.

**Why:** Minimizes blast radius. If training produces a bad head, rolling back is one resource swap.

## Risks / Trade-offs

- **Short keyword + common phonemes** → "Вега" is two short syllables and rhymes with several common Russian words. **Mitigation:** explicit near-miss negatives during training; calibration report measures FPR per hour on real audio before we ship a default threshold.
- **One-user training bias** → head may fail on other speakers. **Mitigation:** synthetic data covers many voices; explicit non-goal for v1; documented in README so future contributors do not expect general speaker coverage.
- **Piper voice quality varies** → some `ru_RU-*` voices sound robotic and may bias the head toward TTS-like cadence. **Mitigation:** use multiple voices, include real recordings, time-shift / pitch augmentation.
- **Embedding model drift** → if upstream openWakeWord updates `embedding_model.onnx` later, all cached embeddings become invalid. **Mitigation:** the cache filename includes the SHA256 of the embedding model; training fails loudly if the bundled model has changed since the cache was built.
- **Cyrillic / file naming** → some Swift Bundle paths historically misbehave with Unicode resource names. **Mitigation:** keep the on-disk filename ASCII (`Vega.onnx`), keep the candidate name ASCII (`"Vega"`), restrict Cyrillic to dataset labels and human-readable text.
- **Dataset reproducibility** → Common Voice and Piper voice releases evolve; "the dataset" changes over time. **Mitigation:** the report records exact dataset versions / Piper voice file hashes; head retrains do not have to match historical numbers but the report explains why.
- **Repo bloat** → committing many head iterations would balloon git. **Mitigation:** only the currently-shipped `Vega.onnx` lives in `Resources/`; experimental heads live under `tools/wake-training/checkpoints/` which is gitignored.

## Migration Plan

1. Land `tools/wake-training/` infrastructure first (scripts, README, requirements lock); no model file, no Swift change yet.
2. Generate a v0 head, run calibration, commit the model file and the calibration report.
3. In the same PR or the next, swap `candidateNames` to `["Vega"]`, delete `Janet.onnx` and `edna.onnx`, update `mac-ear` spec, update default `wakeThreshold` if the calibration report recommends a different value.
4. Verify on-device: run the Mac Ear locally, confirm "Вега" fires within ~400 ms and that a control corpus of speech does not fire over a one-hour soak.

**Rollback:** revert the resource swap commit; the previous `Janet.onnx`/`edna.onnx` heads are restored from git history and detection returns to the pre-change candidate set.

## Open Questions

- Final dataset sizes (positives / negatives / hours-of-background) — locked once we measure how Piper actually performs on Russian and how many recordings the user is willing to produce.
- Whether to ship a parallel English "Vega" head in a follow-up change, and if so, whether to do it as a second `candidateNames` entry or as a separate user preference.
- How the calibration soak test corpus is curated (the user's own speech vs. a public Russian podcast) — decide while writing the eval script.
- Whether `tools/wake-training/` will eventually grow a make/just runner; for now plain Python scripts are enough.
